UniSub 定时任务数据流

4 条定时线 · scheduler._loop 统一驱动 · 全量扫描内嵌 NF 同步

线 1 Emby 扫描 + TMDB 刷新
⏱ 轻量 10min / 全量 60min（auto_sync_enabled）
1 访问 Emby API，同步媒体列表 → emby_cache
2 对缓存条目查询 TMDB API → tmdb_cache
3 （仅全量）触发线 3 的 NF 同步 → 见下方
线 2 自动订阅（发现新内容）
⏱ Cron 表达式（schedule_cron）
1 并发抓取：豆瓣 / Mikan / 猫眼 榜单
2 全局 + 每源过滤（类型/年份/评分）
3 TMDB 搜索（优先）/ NF 搜索（兜底）
4 标题相似度校验 → 查本地 DB 去重
5 新条目 → 调 NF 创建订阅 → 写 subscriptions
线 3 NextFind 双向同步
⏱ 嵌入全量扫描 Step 3 / 手动 POST /api/subscriptions/sync
P0 清理 NF 无元数据异常订阅
P1 NF → 本地：更新状态/缺集数；NF 多余 → 取消
P2 本地 → NF：本地有但 NF 没有 → 推送
P3 电影完成状态（NF is_in_library）
P4 剧集完成状态（emby 集数 vs tmdb 集数）
P5 清理 NF 无效 cancelled 条目
线 4 自动缺集补全（MoviePilot）
⏱ 固定间隔（auto_fill_interval_seconds，默认 30s）· 独立于 auto_sync
1 查找 nf_missing_eps > 0 且未完成的订阅
2 按游标（cursor）轮流取一条
3 调用 MoviePilot 搜索 API 触发下载
4 记录日志，游标前进，下轮处理下一条

⚠️ 关键设计：
线3 不是独立定时器 — 它嵌入在 线1 全量扫描的 Step 3 中（每 60min 跑一次）
线2 目的是「发现新内容」— 只判断 已订阅?/已入库? 不判断缺集
线4 才是「缺集 → 触发下载」的逻辑，独立于 auto_sync_enabled 开关
线1 轻量刷新只补 缺失 TMDB 数据；全量才刷新所有条目
